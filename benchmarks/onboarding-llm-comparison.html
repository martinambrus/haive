Which LLM onboards a legacy codebase best?

The same repository was auto-onboarded 10 times through Haive’s deterministic step pipeline, each run driven by a different LLM. Every run was audited across five correctness dimensions (verified against the real source code) and two cost dimensions (compute time & output tokens).

05Notable defects

Correctness failures

qwen3-coder — promoted the global article “MySQL Best Practices for Redaction System”. A portable cross-project article must never contain the project name. Caught & deleted at the review gate, but the promotion decision was wrong.
gpt-oss:120b — hallucinated a jest-specialist agent (no npm/tests exist) and a fake settings table; misframes HTTP actions as php actions/*.php CLI scripts.
gemma4:31b — never flags the deprecated mysql_* extension (the single most important legacy fact) and omits half the app’s subsystems.

Hygiene / lower-severity

Jest false-positive (injected identically into all 10 by a deterministic heuristic matching FCKeditor/YUI JS): qwen, gpt-oss, minimax, gemma baked “Test frameworks: jest” into AGENTS.md; the other six rejected it.
Sandbox-path leaks (/haive/workdir/…) in qwen + antigravity artifacts — non-portable.
amp — duplicate multilingual skills (one feature, twice) + invented Db methods queryCount/lastQuery.
“MariaDB 10.11” anachronism — sourced from harness metadata; asserted as the app’s DB by qwen/gemma without noting the real mysql_* 3.23-era extension.

06Recommendations

  1. Default / highest-stakes onboarding → Claude Code. Most reliable, no weak dimension, no hallucinations, best judgment. Accept ~55 min compute.
  2. High-throughput / cost-sensitive → deepseek-v4-pro. ~96% of Claude’s correctness at ~half the compute; best detection + agents. Caveat: it promotes nothing to the global KB and generates fewer skills — nudge it on both.
  3. Honorable mentions: kimi-k2.7 (fast + strong, best non-deepseek value) and minimax-m3 (deepest KB + skills, mid speed).
  4. Deprioritize z.ai / glm-4.6: quality is real, but three faster models match or beat it — not worth 2× the compute.
  5. Avoid for production onboarding: qwen, gpt-oss, gemma (hallucinations / shallow output / global-KB contamination).
▸ Methodology & caveats
  • Ground truth. All 10 onboarded the same repo (re-uploaded per run → separate workspaces, so each LLM’s .claude/ artifacts persisted independently). Four parallel auditor agents verified claims against the real source to file:line.
  • Where the data lives. Counts & timing from task_steps / cli_invocations (main DB); generated content from each repo’s on-disk .claude/ dir; global articles from the separate haive_kb_global DB (filtered by source_task_id).
  • Tokens are not comparable across providers. Claude reports cache-inclusive totals (14K input / 6.2M total); ollama-cloud reports raw uncached input (12–18M); antigravity reports 0 (broken tracking). Only output tokens and CLI compute-minutes are used for cost.
  • Wall-clock is misleading. It includes user-idle-at-gate time. deepseek’s 729-min wall was only 28 min compute (the task sat between gates). Speed is ranked by CLI active minutes.
  • Step completion is uniform (~20 done / 2 skipped for all 10) — the pipeline’s stubs/fallbacks mask weak models, so all real signal is in content quality, not completion.
  • Scores are an LLM-assisted structured audit; treat single decimals as indicative, not laboratory-precise. The tiers and per-step winners are robust.
